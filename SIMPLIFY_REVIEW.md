### #4 — `SnippetParamMarker` and `SnippetParamNameCollector` duplicate descent logic

**Level**: 2: within-crate
**Impact**: medium
**Files**: `crates/svelte_analyze/src/passes/template_side_tables.rs`

The call site runs them sequentially over the same AST with a comment noting the need for two passes.

**Proposed change**: Combine into a single visitor that does both operations:
```rust

}
```
The `visit_binding_identifier` marks the getter AND collects the name. One walk instead of two.

**Balance**: Removes ~25 lines and one full AST walk. The two operations are independent (no ordering dependency), making combination straightforward. The comment "must be collected in a separate pass" is incorrect — the name collection doesn't depend on mark_getter results.

### #6 — Two separate `needs_context` aggregation loops in `lib.rs`

**Level**: 2: within-crate
**Impact**: low
**Files**: `crates/svelte_analyze/src/lib.rs`

**Current state**: Lines 153–158 check `info.needs_context` across all expressions. Lines 165–177 check rest-prop references across all expressions. Both iterate `expressions.values().chain(attr_expressions.values())` with an early-exit guard on `data.needs_context`. They can't be trivially merged because the second check depends on `post_resolve` completing (which marks rest_prop symbols), but currently both run after `post_resolve`.

**Proposed change**: Merge the two loops into one:
```rust
if !data.needs_context {
    data.needs_context = data.expressions.values()
        .chain(data.attr_expressions.values())
        .any(|info| {
            info.needs_context
                || (matches!(info.kind, MemberExpression | CallExpression { .. })
                    && info.ref_symbols.iter().any(|&sym| data.scoping.is_rest_prop(sym)))
        });
}
```

**Balance**: Saves one full iteration over all expressions. However, the combined predicate is harder to read than two separate clear checks. Marginal net improvement.
